# Agent Workflow Platform 思路记录

## 1. 基本判断

平台不应该预设一条固定的 RP 管线。

平台更像 ComfyUI：它提供运行环境、节点系统、连线系统、数据流、插件和执行记录。真正的管线应该由用户自己定义。

用户可以根据自己的需求，选择节点、连接节点、决定数据怎么流动。有人想要复杂流程，有人只想直出正文，这两种都应该由同一个平台支持。

## 2. 平台与管线的关系

平台相当于 ComfyUI。

管线相当于用户保存下来的 workflow。

平台不负责规定“正确流程”是什么。平台负责让用户搭出来的流程可以运行、可以观察、可以保存、可以复用。

管线由用户自己连线决定：

```text
节点一 -> 节点二 -> 节点三 -> 输出
```

不同用户可以有完全不同的管线。

## 3. 直出正文的例子

有些用户不需要复杂 Agent 协作，只想直接生成正文。

这种情况下，用户可以自己连一条很短的线：

```text
用户输入
  -> 世界书 / 数据库检索
  -> 正文生成 Agent 节点
  -> 正文输出
```

这里不需要固定的 StateAgent、WorldAgent、MemoryAgent、CriticAgent。用户要不要这些环节，应该由用户自己决定。

## 4. 节点的理解

节点不是简单函数，也不一定是固定角色。

节点可以是一个被打包起来的 Agent。

这个 Agent 内部有：

```text
LLM 内核
当前角色提示词
用户提供的 skill
用户提供的插件
可见的数据输入
输出格式
```

也就是说，节点可以理解为：

> 被用户配置过、被平台包装起来、可以放进 workflow 里使用的 Agent 能力块。

## 5. LLM 与 Agent 的关系

LLM 是 Agent 的内核。

提示词告诉 LLM 当前要做什么。

比如正文生成 Agent：

```text
LLM: pro 模型
提示词: 你负责根据输入上下文生成正文
可见 skill: 用户提供的正文写作 skill
可见插件: 用户允许这个 Agent 使用的插件
输出: 正文内容
```

Agent 不应该默认看见所有 skill 和插件。

一个 Agent 只能看见被授予给它的 skill 和插件。没有授予的能力，对这个 Agent 来说就像不存在。

## 6. Skill / 插件的理解

skill 和插件是 Agent 的工具。

它们不应该全局暴露给所有 Agent。

用户应该可以决定：

```text
这个 Agent 可以使用哪些 skill
这个 Agent 可以使用哪些插件
这个 Agent 不能看见哪些能力
```

例如正文生成 Agent 可以看见写作相关 skill，但不应该看见修改世界状态、写入长期记忆、操作数据库结构之类的插件。

这不是靠提示词告诉它“不要用”，而是让它根本看不见。

## 7. 数据线的理解

像 ComfyUI 一样，不同数据应该有不同类型。

不同类型的数据线可以用不同颜色表达。

例如：

```text
用户输入
世界书内容
数据库检索结果
角色状态
记忆
正文草稿
最终正文
调试信息
```

不同节点的输入输出口应该声明自己接受什么数据、输出什么数据。

用户连线时，实际上是在决定数据怎么流动。

## 8. Agent 不应该一开始被规定死

Agent 不应该一开始就被强行规定成：

```text
WorldAgent 只能做世界
MemoryAgent 只能做记忆
WriterAgent 只能写正文
```

更自然的方式是：

```text
用户创建一个 Agent 节点
给它选择模型
给它写提示词
给它分配 skill / 插件
给它连接输入数据
定义它输出什么
```

这样 Agent 是用户搭建 workflow 时使用的一种智能节点，而不是平台预设好的固定岗位。

## 9. 平台真正应该提供的东西

平台提供的是工作台，而不是固定流程。

平台应该提供：

```text
节点
连线
数据类型
workflow 保存和加载
Agent 节点包装
skill / 插件分配
运行记录
结果输出
```

用户负责决定工作流怎么搭。

平台负责让这个工作流能运行。

## 10. 核心一句话

平台像 ComfyUI。

管线是用户自己连出来的 workflow。

节点是被打包起来的能力块，其中一种重要节点就是 Agent。

Agent 以 LLM 为内核，通过提示词知道当前任务，通过用户授予的 skill 和插件完成任务。

Agent 只能看见被分配给它的能力，看不见其它无关能力。

## 11. 多 Agent 与缓存成本

Agent 不是平台里的单例，也不是一组被平台预设死的固定岗位。

用户可以在同一个 workflow 里创建多个 Agent 节点。每个 Agent 节点都是独立实例，有自己的模型、提示词、skill、插件权限、输入端口、输出格式和缓存策略。

如果多个 Agent 节点之间没有数据依赖，它们应该可以并行执行。平台只根据连线和数据依赖决定执行顺序，而不是根据 Agent 名称或角色决定执行顺序。

由于缓存命中价格很低，Agent 节点在组织上下文时应该注意缓存友好：

```text
稳定内容放前面：
平台级执行协议
Agent 系统提示词
被授权的 skill 内容
被授权的插件说明
稳定世界观 / 角色设定
workflow 节点配置摘要

动态内容放后面：
用户本次输入
上游节点本次输出
运行时临时状态
当前输出格式要求
```

每个 Agent 节点应该独立计算自己的可缓存上下文前缀，避免不同 Agent 互相污染缓存。运行记录里也应该尽量记录模型、token 使用、缓存命中信息和上下文 hash，方便后续优化成本。
