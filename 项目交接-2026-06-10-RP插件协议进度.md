## 1. 当前项目定位

项目现在是一个面向 Agent/RP 角色扮演的类 ComfyUI 工作流平台。短期重点不是完整复刻 ComfyUI，而是把 RP 工作流跑通，并为后续持续增加节点、Agent skill、插件、外部模型节点打好基础。

当前核心方向：

- 以节点图方式编排 Agent/RP 工作流。
- 节点不再只作为前端硬编码按钮存在，而是逐步转为运行时节点目录。
- 工作流模板不硬编码铺在页面上，而是通过模板系统加载。
- RP 场景优先：用户输入、世界书检索、记忆召回、角色卡、场景状态、RP 生成、连续性检查、文本输出。
- 插件协议开始成型：`node.plugin.json + executor adapter + permissions + dependencies + compatibility`。

## 2. 已完成的关键能力

### 2.1 RP 关键节点

当前已经具备一组 RP 核心节点：

- `worldbookSearch`：世界书检索，读取 `data/worldbook.json`，返回命中的设定上下文。
- `memoryRecall`：记忆召回，读取 `data/memories.json`，返回长期记忆、关系状态、玩家偏好、未解决伏笔等。
- `rpCharacterCard`：角色卡节点，输出 `character_profile`。
- `rpSceneState`：场景状态节点，输出 `scene_state`。
- `rpDialogueDirector`：RP 对话导演节点，整合玩家输入、角色、场景、世界书、记忆库，生成 RP 回复。
- `rpContinuityCheck`：连续性检查节点，检查人设、世界观、关系连续性、玩家行动权。
- `textOutput`：最终文本输出。

这些节点已经能组成一条完整 RP 工作流。

### 2.2 RP 参考工作流

### 2.4 节点输出预览与运行快照

前端节点检查器已经能展示：

- 选中节点的基础定义。
- 节点 preview 说明。
- 最近一次运行输入快照。
- 最近一次运行输出快照。
- 节点 metadata，例如世界书命中条目、记忆命中条目、模型调用信息。
- 运行日志点击后可以定位到对应节点。

这对 RP 很重要，因为用户需要知道：

- 世界书命中了哪些设定。
- 记忆库召回了哪些内容。
- Agent 收到了什么上下文。
- 最终 RP 输出来自哪条链路。

## 3. 插件节点协议当前进度

### 3.1 新增插件 SDK 协议

位置：`packages/plugin-sdk/src/index.ts`

已新增：

- `NodePluginManifest`
- `PluginPermission`
- `PluginDependency`
- `PluginCompatibility`
- `NodePluginRuntimeContext`
- `NodeExecutorFactory`
- `validateNodePluginManifest`

manifest 关键字段：

```json
{
  "schemaVersion": 1,
  "id": "awp.rp-core",
  "label": "RP Core Nodes",
  "version": "0.1.0",
  "enabled": true,
  "compatibility": {
    "app": ">=0.1.0",
    "workflowSchema": 1
  },
  "permissions": ["worldbook:read", "memory:read", "model:call"],
  "dependencies": [],
  "executor": {
    "adapter": "local-module",
    "entry": "./executor.mjs"
  },
  "nodes": []
}
```

当前支持的权限枚举：

- `filesystem:read`
- `filesystem:write`
- `network`
- `model:call`
- `memory:read`
- `memory:write`
- `worldbook:read`
- `worldbook:write`

注意：权限目前已经进入 manifest 协议，但还没有做强制权限沙箱，只是为后续插件管理、审批和安全隔离打基础。

### 3.2 新增 RP 插件包

位置：

- `plugins/rp-core/node.plugin.json`
- `plugins/rp-core/executor.mjs`

`rp-core` 是当前第一个真正插件包样例。它把 RP 节点从“内置节点定义”推进到“外部插件 manifest + executor adapter”的形式。

插件声明的节点：

- `worldbookSearch`
- `memoryRecall`
- `rpCharacterCard`
- `rpSceneState`
- `rpDialogueDirector`
- `rpContinuityCheck`

插件 executor 通过受控 context 访问能力：

- `readWorldbook`
- `readMemories`
- `rankEntries`
- `serializeEntries`
- `executeAgent`
- `onToken`

这样 executor 不需要直接知道服务端文件路径和模型细节，后续也更容易加权限控制。

### 3.3 服务端插件加载

位置：`apps/web/scripts/serve.mjs`

服务端启动时会：

- 扫描 `plugins/*/node.plugin.json`
- 读取并校验 manifest
- 跳过无效插件
- 跳过 `enabled: false` 的插件
- 合并 `nodeRegistry + pluginCatalog`
- 动态 import 本地 executor adapter
- 让插件 executor 覆盖同名内置 executor

当前插件目录：

```text
plugins/
  rp-core/
    node.plugin.json
    executor.mjs
```

### 3.4 API-first 节点目录

已实现：

- `GET /api/nodes`
- `GET /api/templates`
- `POST /api/workflows/validate`
- `POST /api/run-workflow`
- `POST /api/run-workflow-stream`

`GET /api/nodes` 现在返回：

- 合并后的节点列表。

这意味着前端节点库可以从 API 获取当前可用节点，而不是只依赖编译期 TypeScript registry。

## 8. 当前仍然不足

### 8.1 插件系统还不是完整生命周期

已完成 manifest、executor adapter、动态发现、运行时 catalog，但还缺：

- 插件安装 API。
- 插件启用/禁用 API。
- 插件管理前端页面。
- 版本兼容性强校验。
- 依赖解析。
- 权限审批与权限强制执行。
- 插件运行隔离或沙箱。
- 远程 HTTP executor 实际实现。

### 8.2 节点参数 schema 还偏基础

已有 `configFields`，但还缺：

- `boolean`
- `multiselect`
- `json`
- `file`
- `secret`
- `model`
- 条件字段
- 参数 presets
- 高级参数折叠
- 字段级校验错误
- 节点帮助文档

### 8.3 RP Agent 还需要更强的可观测性

虽然已有输入/输出/metadata 快照，但还需要：

- Prompt assembly 展开查看。
- token 估算。
- 模型名、耗时、成本展示。
- 世界书命中详情展开。
- 记忆命中详情展开。
- 中间结果复制按钮。
- 每个节点失败定位到字段或输入。

### 8.4 工作流运行调度仍偏基础

当前是 DAG batch 执行。后续 Agent/RP 场景还需要：

- 从某个节点重新运行。
- 只运行选中节点或下游节点。
- 节点级缓存。
- 取消、暂停、重试。
- 可恢复 run session。
- human-in-the-loop 审批节点。

### 8.5 画布体验还不够 ComfyUI 化

后续可以补：

- 节点搜索。
- 右键菜单按分类添加节点。
- 框选、多选、批量删除。
- 复制/粘贴子图。
- 节点分组与注释框。
- 节点折叠。
- 子图/宏节点。
- 自动布局。
- 对齐和吸附。

## 9. 建议下一步优先级

建议下一轮优先做这几件，顺序不要太散：

1. 插件管理 API：`GET /api/plugins`、`POST /api/plugins/:id/enable`、`POST /api/plugins/:id/disable`。
2. 前端插件管理面板：展示插件版本、权限、依赖、节点列表、启用状态。
3. RP 节点详情预览增强：世界书命中、记忆命中、prompt assembly、最终输出一屏看清。
4. 节点参数 schema 升级：先补 `boolean / json / secret / model / multiselect`。
5. 为 `remote-http` executor 做一个最小实现，让外部服务也能作为节点插件接入。

如果时间仍然紧，优先级最高的是第 1、2、3 项。它们直接影响后续加节点和 RP 使用体验。

## 10. 给下一位开发者的注意事项

- 不要再把新节点只写在前端页面里。
- 新节点优先走 `plugins/<plugin-id>/node.plugin.json`。
- 需要执行逻辑时，在同目录提供 `executor.mjs`。
- 如果只是临时内置节点，可以继续放 `packages/workflow-core/src/nodeRegistry.ts`，但要明确它是 fallback 或 core node。
- 改动节点端口、数据类型时，必须同时检查连线规则、校验、前端端口渲染。
- 完成后至少跑：

```powershell
npm run test
npm run verify
npm run build
```

如果改前端交互，还要启动：

```powershell
npm run serve
```

并在浏览器中检查：

- 节点列表是否能从 API 加载。
- RP 模板是否能加载。
- 世界书与记忆库是否能被命中。
- 节点检查器是否能看到输入/输出/metadata。
- 插件节点是否能正常校验和执行。
