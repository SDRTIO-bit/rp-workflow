## A. Current Real Data Flow

### A.1 Official RP request → Workflow → Writer (verified)

The production RP workflow is bound to `POST /api/rp` and lives at
`data/workflows/rp-unified-stateful-production-v1.json` (git-ignored `data/`
dir). The call chain, with file:line:

```
POST /api/rp  { sessionId, turnId, userInput, worldbook.resourceRef, memory.namespace, preset.text? }
  │
  ▼  apps/server/src/routes/rp.ts:22  (createRpRoutes, mounted at composition.ts:447)
  │

  ▼  Workflow node DAG (per project-state-through-p15.2.md §4):
        agentSessionLoadV1 → agentSessionLastAssistantOutput

**Worldbook data reaches the Writer through workflow edges**, not direct
injection: `dynamicWorldbook` node loads entries from the store under
`session:{sessionId}:{resourceRef}` → entries flow into the lore retriever →
the context assembler builds a `[World & Character Lore]` section
(`rp-runtime/src/nodes/rpContextAssemblerV1.ts:300` `buildLoreSection`) → that
assembled context becomes the Writer's `context` input port.

### A.2 Dynamic Worldbook Store (verified)

- **Interface**: `DynamicWorldbookStore` at
  `packages/workflow-worldbook/src/types.ts:116-136` — `load(scopeKey,
resourceRef)`, `save(scopeKey, resourceRef, snapshot)`, plus
  operation-record methods.
- **Entry shape**: `DynamicWorldbookEntryV1` at `types.ts:11-22` —
  `{ id, content, title?, type?, tags?, entityIds?, priority?, metadata?,
createdAt?, updatedAt? }`. `id` and `content` are required non-empty
  strings (≤10k chars). Normalized by `normalizeEntry()`
  (`normalize.ts:27-159`).
- **Only implementation**: `InMemoryDynamicWorldbookStore`
  (`inMemoryStore.ts:15`) — `Map<StoreScope, Snapshot>`. **No file/DB
  backend. Server restart loses all worldbook data.**
- **Scope key**: `buildScopeKey()` at `operations.ts:674-689` —
  `session:{sessionId}:{resourceRef}` for `"session"` lifecycle (production
  default). `"project"` / `"persistent"` lifecycles are defined but rejected
  at `operations.ts:81-84`.
- **Registration path**: no `registerWorldbook()` function. Entries enter via
  (a) `seedSessionWorldbookIfEmpty()` bulk seed from `data/worldbook.json`
  (`officialRpService.ts:213-237`), or (b) runtime `append`/`upsert`/`replace`
  operations through the `dynamicWorldbook` node executor. Card import can
  call `worldbookStore.save(scopeKey, resourceRef, snapshot)` directly.

### A.3 Legacy Worldbook shape (verified)

`rp-runtime/src/worldbook/types.ts:35-68` defines `WorldbookEntryV1` with the
SillyTavern-compatible fields: `keys`, `secondaryKeys`, `aliases`,
`selective`, `category`, `priority`, `depth`, `relatedEntryIds`,
`excludesEntryIds`, `visibility`, `constant`. The legacy
`rpWorldbookRetrieverV1` (`rp-runtime/src/worldbook/rpWorldbookRetrieverV1.ts`)
consumes this shape via a `worldbookEntries` input port. **The production
workflow uses `dynamicWorldbook + genericRetriever`** (per project-state §4),
which targets the simpler `DynamicWorldbookEntryV1` (tags/entityIds matching),
not the legacy retriever. This shape gap is central to the worldbook mapping
design (§G).

### A.4 Agent Session (verified)

- **No `createSession` API.** Sessions are implicit: `AgentSessionStore.load()`
  returns null for unknown keys; `append()`/`commitIdempotent()` creates
  lazily. Interface at `packages/agent-runtime/src/agentSessionStore.ts:43-63`.
- **Session key**: `AgentSessionKeyV1` at `agentSession.ts:16-22` —
  `{ tenantId, workflowInstanceId, conversationId, agentNodeId, branchId? }`.
  In production RP, `conversationId = request.sessionId`,
  `tenantId="default"`, `workflowInstanceId="rp-prod-1"`,
  `agentNodeId="writer-main"` (set at `officialRpInputAdapter.ts:124-135`).
- **Implementations**: `InMemoryAgentSessionStore` (`agentSessionStore.ts:94`)
  and `FileAgentSessionStore` (`agentSessionStore.ts:151`, atomic temp+rename
  to `agent-sessions.json`). Selection via env `AGENT_SESSION_STORE`
  (`env.ts:44`).
- **No greeting/seed concept.** `AgentSessionContextV1` has no
  `greeting`/`initialAssistantTurn`/`isSeed` field. `AgentTurnV1` has
  `turnIndex`, `input`, `assistantOutput`, `modelConfig`, `tokenUsage`,
  `createdAt` — no greeting flag. The only precedent for seeding a turn is
  in tests (`rpNoveltyReplayE2E.test.ts:379-391`) which call `store.append()`
  directly.
- **turnId**: exists only at the API/client/dedup layer
  (`OfficialRpRequestV1.turnId`, `AgentSessionCommitDedupKeyV1`); the
  persisted `AgentTurnV1` uses `turnIndex` (1-based, assigned at
  `agentV2.ts:278` as `turns.length + 1`). Dedup key =
  `(sessionId, agentNodeId, turnId)` + `contentHash`
  (`agentSessionStore.ts:24-31`): same turnId+hash = silent skip; same
  turnId+different hash = 409.

### A.5 Memory & Worldbook namespace isolation (verified)

- **Memory**: `WorkflowMemoryStore` (`workflow-memory/src/types.ts:81-102`)
  partitions by `namespace: string`. `InMemoryWorkflowMemoryStore` uses
  `Map<namespace, Map<id, record>>` (`inMemoryStore.ts:13`);
  `FileWorkflowMemoryStore` uses `Record<namespace, Record<id, record>>`
  (`fileStore.ts:19`). Two sessions sharing a namespace **collide** (overwrite
  by id, cross-read on list). Web client avoids this by generating
  `rp-session:${uuid}` per session (`rpSessionState.ts:43`).
- **Worldbook**: scoped by composite key `(scopeKey, resourceRef)` where
  `scopeKey = session:${sessionId}:${resourceRef}`. Two sessions with
  different sessionIds are isolated even with the same resourceRef.

@awp/workflow-stdlib      → workflow-core

@awp/rp-runtime           → workflow-core, agent-runtime, plugin-sdk

                            workflow-retrieval, workflow-memory directly — but
                            tsconfig references them)
@awp/web (app)            → agent-runtime, memory-core, plugin-sdk, workflow-core
```

Build system: **npm workspaces** (`package.json:6` `npm@11.6.1`,
`workspaces: ["apps/*","packages/*"]`), TypeScript composite projects via
`tsc -b`. Tests: Vitest, co-located `*.test.ts`.

### C.2 Decision: new package `@awp/card-import` under `packages/card-import/`

**Rationale**:

1. **Precedent**: `rp-runtime` was created as a new package for a substantial
   new domain. Card import has comparable complexity (SillyTavern V3 parsing,
   hash dedup, manifest, greeting extraction, worldbook mapping, blocked-feature
   detection).
2. **Dependency direction is clean**: `card-import` depends on
   `@awp/workflow-worldbook` (for `DynamicWorldbookStore`,
   `DynamicWorldbookEntryV1`, `normalizeEntry`) and `@awp/workflow-core` (for
   shared types). It does NOT depend on `rp-runtime`, `agent-runtime`, or
   `apps/server` — keeping the import pipeline RP-agnostic and reusable for
   future non-RP products.
3. **Server consumes it**: `apps/server` adds `@awp/card-import` as a dep and
   wires routes/services in `apps/server/src/routes/cards.ts` +
   `apps/server/src/services/cardImportService.ts` (the route + I/O orchestration
   layer stays in the server, matching the pattern where `rp-runtime` logic is
   in the package but `officialRpService` + `routes/rp.ts` are in the server).
4. **Testability**: package-level Vitest with co-located `*.test.ts`, fixtures
   in `packages/card-import/src/__fixtures__/` (desensitized, tiny).

**What goes in `@awp/card-import`** (pure logic, no Node HTTP, no Hono):

- SillyTavern V3 JSON parsing + structural validation
- Content hash (SHA-256) computation
- `CardManifestV1` builder
- Greeting extraction + cleaning
- `character_book.entries` → `DynamicWorldbookEntryV1` mapping
- Blocked-feature / variable-condition detection (metadata only, no execution)
- Card Store interface + file-backed implementation (atomic writes)
- All data-contract types (§D)

**What stays in `apps/server`**:

- `routes/cards.ts` — Hono route handlers (multipart parse, JSON responses)
- `services/cardImportService.ts` — orchestrates: parse file → call
  `card-import` package → register in `worldbookStore` → write files
- Composition wiring in `composition.ts` (new `cardsDir` env, store instance,
  route registration)
- Card-aware worldbook seeding integration in `officialRpService.ts`

**What stays in `apps/web`**:

- Card upload UI component + card list / greeting selection page

### C.3 Anti-pattern: do NOT route cards through `rpCharacterCard` plugin node

A plugin node `rpCharacterCard` exists at
`plugins/rp-core/node.plugin.json`. It takes character config (`name`,
`persona`, `voice`, `boundaries`) as **node parameters** in workflow JSON.
**Imported card data must NOT be mapped into this node.** Doing so would embed
external card content into the workflow JSON / node config — violating the
hard constraint that character cards are external resources and must not
enter Workflow core, Agent Profile, System Prompt, or RP Runtime. The
`rpCharacterCard` node is for hand-authored workflow-level character config,
not for imported external cards. Card data enters solely via
`worldbook.resourceRef` → Dynamic Worldbook → Retrieval → Writer Context.

All types live in `@awp/card-import` and are exported from its package root.
Field visibility is tagged per field: **[W]** Web-visible, **[S]** Server-only,
**[P]** Persistable (written to disk), **[X]** Must NOT enter Prompt,
**[R]** Must NOT enter API Response.

### D.2 CardImportRequestV1 (multipart input convention)

No TypeScript body type — this is a `multipart/form-data` convention parsed
via Hono's `c.req.parseBody()`:

```
POST /api/cards/import
Content-Type: multipart/form-data; boundary=...

  field "file"      : exactly one file part, .json, UTF-8, ≤ MAX_CARD_BYTES
  field "filename"  : optional text, original filename (server also reads
                      part.filename; this is a fallback)
  field "overwrite" : optional text, "true"|"false" (default false). If true
                      and a different-content file with same sourceFilename
                      exists, still creates a new cardId (hash differs). This
                      flag only controls whether an identical-hash re-upload
                      returns the existing cardId (false) or errors (true).
```

Validation: exactly one file part; filename sanitized via `safeFilename()`;
content-type must be `application/json` or absent; size enforced before parse.

### D.5 ImportedWorldbookEntryV1

This is the mapped entry stored in the card's `worldbook.json` and loaded
into the `DynamicWorldbookStore`. It is a `DynamicWorldbookEntryV1` with
preserved SillyTavern metadata. The base type is
`DynamicWorldbookEntryV1` from `@awp/workflow-worldbook`; this interface
constrains the `metadata` shape:

```typescript
export interface ImportedWorldbookEntryMetadata {
  // Preserved SillyTavern origin (never executed, never auto-injected)
  sourceEntryUid: number | string; // [S][P][X] ST entry uid
  sourceKeys: string[]; // [S][P][X] original keys
  sourceSecondaryKeys: string[]; // [S][P][X]
  sourceConstant: boolean; // [S][P][X] ST constant flag
  sourceSelective: boolean; // [S][P][X]
  sourcePosition: string | null; // [S][P][X] ST position
  sourceDepth: number | null; // [S][P][X]
  sourceProbability: number | null; // [S][P][X]
  sourceGroup: string | null; // [S][P][X]
  sourcePreventRecursion: boolean; // [S][P][X]
  sourceExtensions: Record<string, unknown> | null; // [S][P][X]
  sourceEnabled: boolean; // [S][P][X] false = disabled in ST

  // Variable / condition detection (preserved, not executed)
  unsupportedVariableCondition: boolean; // [S][P][X] EJS/getvar/MVU condition present
  variableConditionSource: string | null; // [S][P][X] raw condition text, truncated
  detectedVariableRefs: string[]; // [S][P][X] variable names referenced

  // Card provenance
  cardId: string; // [S][P][X]
  importSchemaVersion: 1; // [S][P][X]
}

// The stored entry is DynamicWorldbookEntryV1 with metadata: ImportedWorldbookEntryMetadata
export type ImportedWorldbookEntryV1 = DynamicWorldbookEntryV1 & {
  metadata: ImportedWorldbookEntryMetadata;
};
```

Field mapping rationale (see §G for the full table): `id` = stable
`card:<cardId>:e<uid>`; `content` = ST `content`; `title` = ST
`comment`/`name`; `tags` = merged `keys` + `secondary_keys` (for
`genericRetriever` keyword matching); `priority` = ST `insertion_order`;
`type` = mapped from ST `category`/`position`. All ST-specific flags
(`constant`, `selective`, `probability`, etc.) are preserved in `metadata`
**but never interpreted at runtime in V1**.

## E. File Directory & Persistence

### E.1 Layout

```
<DATA_DIR>/cards/<cardId>/
  source.json              # immutable original file bytes (UTF-8 JSON), read-only after write
  manifest.json            # CardManifestV1 (paths stripped of [R] fields when served)
  greetings.json           # ImportedGreetingV1[] (cleaned; separated fragments included for server)
  worldbook.json           # ImportedWorldbookEntryV1[] (the mapped entries)
  import-report.json       # { warnings, blockedFeatures, capabilities, generatedAt }
```

`<DATA_DIR>` is the existing `env.dataDir` (`apps/server/src/env.ts:152`,
default `F:/1/新建文件夹 (2)/data`). The `data/` directory is git-ignored
(`.gitignore:7`), satisfying "character cards never enter Git". A new env var
`CARDS_DIR` (default `resolve(env.dataDir, "cards")`) follows the precedent of
`WORKFLOW_MEMORY_DIR` / `AGENT_SESSION_DIR`.

### E.2 cardId generation

`cardId = sha256(sourceFileBytes).toString("hex")` — 64 hex chars, inherently
filename-safe, content-addressed. Same file content → same cardId → dedup.
Different content → different cardId → new card. The `worldbookResourceRef`
is `"card:" + cardId`, which satisfies the `^[a-zA-Z0-9_\-.:]{1,128}$`
validation rule (`officialRpInputAdapter.ts:89-90`).

### E.3 Atomic write

Reuse the `FileWorkflowCheckpointStore` pattern
(`workflow-persistence/src/stores.ts:60-86`):

1. `safeFilename(id)` — strip non-`[a-zA-Z0-9._-]` to `_`, slice to 64 chars
   (prevents directory traversal; cardId is already hex so this is belt-and-
   braces, but `sourceFilename` from the upload needs it).
2. Write to `<file>.tmp`, then `rename(<file>.tmp, <file>)` — atomic on the
   same filesystem.
3. Each of the 5 files is written independently with its own temp+rename.

### E.4 Dedup on same hash

```
on import request:
  rawBytes = read uploaded file
  hash = sha256(rawBytes).hex
  cardDir = resolve(cardsDir, hash)
  if exists(cardDir/manifest.json):
    # dedup hit — same content already imported
    load existing manifest + greetings
    return CardImportResultV1 { cardId: hash, alreadyExisted: true, ... }
    # do NOT re-create dir, do NOT re-register worldbook
  else:
    # new card
    parse → validate → build manifest → extract greetings → map worldbook
    mkdir -p cardDir (atomic: create temp dir, populate, rename)
    write 5 files atomically
    return CardImportResultV1 { cardId: hash, alreadyExisted: false, ... }
```

The entire `cardDir` creation uses a temp-directory-then-rename pattern:
write everything into `<cardsDir>/.<hash>.tmp/`, then
`rename(<cardsDir>/.<hash>.tmp, <cardsDir>/<hash>)`. This makes the card
atomically appear or not appear — no half-written card directory is ever
visible.

### E.5 Corruption handling

- On read (list/get/seed): if any of the 5 files fails JSON parse or schema
  validation, the card is marked `corrupted` in the list response and skipped
  for seeding. The corrupt files are NOT deleted automatically (preserves
  evidence); a future delete API or manual cleanup handles them.
- `source.json` is never rewritten after first write (append-only / immutable).
  If `source.json` is missing but other files exist, the card is corrupt
  (hash cannot be verified).

### E.6 Import interrupt cleanup

Temp directory `<cardsDir>/.<hash>.tmp/` is cleaned up on any error during
import (parse failure, validation failure, write failure). A
`finally`-style guard removes the temp dir if it still exists. On server
restart, orphaned `.tmp` dirs are swept at startup (best-effort `readdir` +
`rmdir` for entries matching `/.+\.\w+\.tmp$/` ... actually the pattern is
`/^\.[0-9a-f]{64}\.tmp$/`).

### E.7 Filename / path traversal

- `cardId` is hex (64 chars) — safe for directory name.
- `sourceFilename` from upload is sanitized via `safeFilename()` before
  storing in `manifest.sourceFilename`. It is stored as metadata only; it is
  never used to construct a filesystem path. The on-disk file is always
  `source.json` (fixed name), so the upload filename cannot affect the path.

### E.8 Limits (configurable via env)

| Limit                      | Env var                     | Default | Rationale                                                                                           |
| -------------------------- | --------------------------- | ------- | --------------------------------------------------------------------------------------------------- |
| Max file size              | `MAX_CARD_BYTES`            | 10 MB   | SillyTavern cards with embedded images can be large; JSON-only cards are <1 MB. 10 MB is generous.  |
| Max JSON depth             | `MAX_CARD_JSON_DEPTH`       | 64      | Defense against pathological nesting.                                                               |
| Max worldbook entries      | `MAX_WORLDBOOK_ENTRIES`     | 2000    | Test card has 40; 2000 is headroom.                                                                 |
| Max greetings              | `MAX_GREETINGS`             | 64      | Test card has 6; 64 is headroom.                                                                    |
| Max entry content length   | `MAX_ENTRY_CONTENT_CHARS`   | 100000  | `DynamicWorldbookEntryV1` already caps at 10000; imported entries are truncated to the store limit. |
| Max source filename length | `MAX_SOURCE_FILENAME_CHARS` | 255     | Filesystem limit.                                                                                   |

Enforced before/at parse time. A limit violation produces a
`CardImportWarningV1` with severity `error` and aborts the import (temp dir
cleaned up).

### E.9 schemaVersion & upgrade

Every `manifest.json` and `import-report.json` carries `schemaVersion: 1`
and `importSchemaVersion: 1`. On read, if `schemaVersion > 1` (future), the
loader rejects with "unsupported manifest schema" (forward-compatible
refusal). If `schemaVersion < 1`, no upgrade is needed (V1 is the first).
Future V2 manifests will include an `upgradeFrom` migration in the
`@awp/card-import` package. Old sessions never auto-migrate to new cardIds
(different content → different hash → different cardId by construction).

### E.10 source.json immutability

After the initial atomic write, `source.json` is opened read-only for all
subsequent operations. The card import service verifies
`sha256(source.json) === cardId` on every read. If the hash mismatches, the
card is marked corrupt. No API in V1 writes to `source.json`.

## F. Import Flow (full closed loop)

```
1. User uploads SillyTavern JSON via Web (multipart/form-data)
2. Server: POST /api/cards/import
   a. Hono c.req.parseBody() → extract "file" part
   b. Size check (≤ MAX_CARD_BYTES) BEFORE parsing
   c. sha256(rawBytes) → cardId
   d. If <cardsDir>/<cardId>/manifest.json exists → dedup: return existing
   e. Parse JSON (depth-limited, entry-count-limited)
   f. Validate: spec === "chara_card_v3" (reject non-V3)
   g. Detect blocked features (JS/eval/EJS/getvar/scripts/remote URLs)
   h. Detect variable system (MVU/Zod schema/initial state/JSON Patch)
   i. Extract greetings (alternate_greetings; skip first_mes as remote placeholder)
   j. Clean greetings (remove status bar placeholders, separate variable tags,
      remove/flag remote content, detect unapplied JSON Patch)
   k. Map character_book.entries → ImportedWorldbookEntryV1[]
      (skip disabled entries; preserve constant/selective in metadata;
      flag variable-condition entries as unsupported)
   l. Build CardManifestV1 + CardCapabilitiesV1 + warnings + blockedFeatures
   m. Atomic write: temp dir → 5 files → rename to <cardsDir>/<cardId>/
   n. Register in DynamicWorldbookStore:
      - For each session that uses this card, seed at session-init time
        (NOT at import time — import only writes files; session init seeds
         the in-memory store under session scope)
   o. Return CardImportResultV1 (paths stripped)

3. Web: display import result (warnings, blocked features, greeting count)
4. Web: list cards (GET /api/cards) → CardListItemV1[]
5. Web: select card → select greeting (GET /api/cards/:cardId/greetings)
6. Web: POST /api/cards/sessions (CardSessionInitRequestV1)
   → Server seeds worldbook store under session:<sessionId>:card:<cardId>
   → Server appends greeting as turn 0 to AgentSessionStore
   → Returns CardSessionInitResultV1 { worldbookResourceRef, memoryNamespace }
7. Web: display the greeting as the opening message (from greeting content,
   not re-generated)
8. Web: subsequent turns call POST /api/rp with:
     { sessionId, turnId, userInput,
       worldbook: { resourceRef: "card:<cardId>" },
       memory: { namespace: "rp-session:<sessionId>" },
       preset: { text: ... } }
   → Existing RP pipeline runs unchanged
```

**Key invariant**: import writes files only; session init seeds the in-memory
worldbook store. This separation means import is restart-safe (files persist)
and the in-memory store is always rehydrated from files at session init,
solving the "worldbook store is in-memory" problem (B6) without implementing
a full file-backed `DynamicWorldbookStore`.

## G. Worldbook Mapping Table

SillyTavern Chara Card V3 `character_book.entries[]` → `ImportedWorldbookEntryV1`
(which is `DynamicWorldbookEntryV1` + `ImportedWorldbookEntryMetadata`).

### G.1 Field mapping

| SillyTavern V3 field            | ImportedWorldbookEntryV1 field                     | Notes                                                                                                                                                                           |
| ------------------------------- | -------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `uid` (or `id`)                 | `id` = `"card:<cardId>:e<uid>"`                    | Stable, namespaced, unique.                                                                                                                                                     |
| `comment` / `name`              | `title`                                            | Truncated to 200 chars.                                                                                                                                                         |
| `content`                       | `content`                                          | Truncated to `MAX_ENTRY_CONTENT_CHARS` (then to store's 10000 limit at seed time).                                                                                              |
| `keys`                          | `tags` (merged)                                    | Also preserved verbatim in `metadata.sourceKeys`.                                                                                                                               |
| `secondary_keys`                | `tags` (merged)                                    | Also in `metadata.sourceSecondaryKeys`.                                                                                                                                         |
| (derived)                       | `type`                                             | Mapped from ST `category` if present, else from `position`.                                                                                                                     |
| `insertion_order` / `priority`  | `priority`                                         | Direct numeric copy.                                                                                                                                                            |
| `constant`                      | `metadata.sourceConstant`                          | **NOT interpreted at runtime in V1.** Not auto-injected.                                                                                                                        |
| `selective`                     | `metadata.sourceSelective`                         | Preserved, not interpreted.                                                                                                                                                     |
| `position`                      | `metadata.sourcePosition`                          | Preserved.                                                                                                                                                                      |
| `depth`                         | `metadata.sourceDepth`                             | Preserved.                                                                                                                                                                      |
| `probability`                   | `metadata.sourceProbability`                       | Preserved, not interpreted (V1 ignores probability gating).                                                                                                                     |
| `group`                         | `metadata.sourceGroup`                             | Preserved.                                                                                                                                                                      |
| `prevent_recursion`             | `metadata.sourcePreventRecursion`                  | Preserved.                                                                                                                                                                      |
| `extensions`                    | `metadata.sourceExtensions`                        | Preserved as-is (may contain variable schema).                                                                                                                                  |
| `disable` / `enabled === false` | **Entry is SKIPPED** (not imported into worldbook) | Counted in `worldbookDisabledCount`; warning `disabled-entries-skipped`.                                                                                                        |
| (variable condition in content) | `metadata.unsupportedVariableCondition = true`     | EJS `<% %>`, `{{getvar:...}}`, `{{setvar:...}}`, MVU tags detected. Condition text truncated in `metadata.variableConditionSource`. **NOT executed. NOT treated as always-on.** |
| (entry references variables)    | `metadata.detectedVariableRefs`                    | Variable names extracted from condition text.                                                                                                                                   |

### G.2 Constant entries — behavior difference (critical)

SillyTavern semantics: `constant: true` → entry is **always** injected into
context regardless of keyword match. The task explicitly requires: "请特别避免
将所有 constant 条目无条件拼进每轮 Prompt."

**V1 design**: constant entries are mapped into the worldbook store with
`metadata.sourceConstant = true` and their `keys` merged into `tags`. They
are retrieved by the existing `genericRetriever` keyword/tag matching — they
surface **only when their keys/tags match user input**. There is **no
always-on injection path** in V1.

This is a **documented behavior difference** from raw SillyTavern, recorded as
warning `constant-entries-not-auto-injected` (severity `warn`, count =
`worldbookConstantCount`). The test card has 26 constant entries; under V1
they will not all appear in every prompt. A future phase may tune the
workflow's `dynamicWorldbook` query command to always-include entries tagged
`metadata.sourceConstant === true` — but that is a **workflow-level
parameter change**, not a code change, and explicitly out of scope for
P-15.3A.

### G.3 Variable-condition entries — preserved, not executed

Entries whose `content` contains EJS (`<% %>`), `{{getvar:...}}`,
`{{setvar:...}}`, or MVU `{{var::...}}` patterns are:

- Mapped with `metadata.unsupportedVariableCondition = true`.
- Their raw condition text is preserved (truncated) in
  `metadata.variableConditionSource`.
- **NOT executed** — the content is stored as-is (the condition tags remain
  as literal text in `content`). This means the Writer may see literal
  `{{getvar:foo}}` text if the entry is retrieved. To avoid confusing the
  model, such entries have their condition tags **wrapped in a comment
  marker**: `<!-- unsupported-variable-condition: {{getvar:foo}} -->`. The
  entry is still retrievable but the model sees a comment, not executed
  output.
- **NOT treated as always-on** — they follow the same keyword/tag retrieval
  as other entries.
- Recorded as warning `entry-variable-condition-unsupported` per entry.

### G.4 No shortcut to Writer Prompt

All entries — constant, non-constant, variable-conditioned — go through the
same path: `CardImport → worldbook.json file → session-init seed →
DynamicWorldbookStore → dynamicWorldbook node → genericRetriever →
rpContextAssemblerV1 → Writer context`. No entry is ever injected directly
into the Writer prompt, `preset.text`, or `System Prompt`. This satisfies
constraint 4.

## H. Greeting Initialization Flow

### H.1 Greeting extraction

The test card's `first_mes` is a **remote opening-page placeholder** (not
usable as a narrative greeting). The 6 `alternate_greetings` are the usable
narrative openings.

**Extraction rules**:

1. `alternate_greetings[]` → each becomes an `ImportedGreetingV1` with
   `greetingId = "g<index>"` (g0..g5), `index` = 0-based.
2. `first_mes` → recorded as warning `remote-first-mes` (severity `warn`),
   NOT included in the greeting list. Its content is not stored in
   `greetings.json` (only the warning references it).
3. If `alternate_greetings` is empty, `defaultGreetingId = null` and a
   warning is emitted (no usable greeting).

### H.2 Default greeting selection

`defaultGreetingId = "g0"` (first alternate_greeting) unless:

- g0 has `hadRemoteContent = true` and another greeting does not → pick the
  first greeting with `hadRemoteContent = false`.
- All greetings have remote content → still pick g0, emit warning
  `greeting-remote-content-removed` for all.

### H.3 Greeting content cleaning

For each greeting, in order:

1. **Status bar placeholder removal**: detect `{{status_bar}}`,
   `{{stats}}`, `{{status_xxx}}`, `{{getvar:status_*}}` patterns; remove
   them; set `hadStatusBarPlaceholder = true`; add to
   `removedFragmentSummary`.
2. **Variable update tag separation**: detect `{{setvar:...}}`,
   `{{addvar:...}}`, `{{incvar:...}}`, `{{decvar:...}}`, MVU
   `{{var::set:...}}` patterns; **remove from content**, store raw tags in
   `separatedVariableTags`; set `hadVariableUpdateTags = true`. These are
   NOT applied (V1 has no variable runtime).
3. **Remote content removal**: detect `<script src=...>`, `<img src=http...>`,
   `<iframe>`, `import(...)`, `fetch(`, jQuery `$.load(...)` patterns;
   remove the tag; store the URL/ref in `separatedRemoteRefs`; set
   `hadRemoteContent = true`.
4. **JSON Patch detection**: detect JSON Patch payloads (e.g.
   `{"op":"replace","path":...}` blocks or `{{patch:...}}` tags); do NOT
   apply; set `hadUnappliedInitialPatch = true`; emit warning
   `greeting-initial-patch-not-applied`.
5. **Whitespace normalization**: collapse excessive blank lines; trim.

The cleaned `content` is what gets committed to the session as the assistant
opening turn. The `separatedVariableTags` and `separatedRemoteRefs` are
persisted in `greetings.json` for future MVU phases but are **never** sent to
the model and **never** in API responses (tagged `[S][P][X][R]`).

### H.4 Greeting enters session (NOT re-generated by Writer)

```
POST /api/cards/sessions  { cardId, greetingId, sessionId, memoryNamespace? }
  │
  ▼  apps/server/src/services/cardImportService.ts (new)
  │   ├─ load manifest + greetings from <cardsDir>/<cardId>/
  │   ├─ validate cardId exists, greetingId valid
  │   ├─ build scopeKey = `session:${sessionId}:card:${cardId}`
  │   ├─ seed DynamicWorldbookStore:
  │   │    store.save(scopeKey, "card:<cardId>", { version:1, entries: worldbookEntries })
  │   ├─ build AgentSessionKeyV1 { tenantId:"default", workflowInstanceId:"rp-prod-1",
  │   │    conversationId: sessionId, agentNodeId:"writer-main" }
  │   ├─ append greeting as turn 0:
  │   │    store.append(sessionKey, {
  │   │      sessionKey,
  │   │      newTurn: { turnIndex: 0, input: "", assistantOutput: cleanedGreeting,
  │   │                 modelConfig: { providerId:"card-import", model:"greeting-seed" },
  │   │                 tokenUsage: { inputTokens:0, outputTokens:0 },
  │   │                 createdAt: new Date().toISOString() }
  │   │    })
  │   └─ return CardSessionInitResultV1
  │
  ▼  Web receives { worldbookResourceRef, memoryNamespace, greetingPreview }
  │
  ▼  Web displays the greeting (from the greeting content it already has)
  │
  ▼  First POST /api/rp { sessionId, turnId:"turn-0001", userInput:"<first user input>",
       worldbook:{ resourceRef:"card:<cardId>" }, memory:{ namespace:"rp-session:<sessionId>" } }
       → existing RP pipeline runs:
         agentSessionLoadV1 loads the session → sees turn 0 (greeting) as history
         agentSessionLastAssistantOutput → reads greeting as "previous assistant output"
         textNoveltyCheck → compares Writer output against greeting (prevents duplicate opening)
         Writer generates turn 1 in response to user input
```

**The greeting is never sent to the Writer for regeneration.** It is
committed as turn 0 via `store.append()` directly. The first `POST /api/rp`
is the user's first real input; the Writer generates turn 1. The greeting
serves as session context and as the novelty-check reference for the first
turn.

### H.5 turnIndex convention

- Greeting = turn 0 (committed at session init).
- First `POST /api/rp` = turn 1 (`turnIndex = turns.length + 1 = 1`).
- `agentSessionLastAssistantOutput` reads the last committed turn = greeting
  at turn 0, so the first novelty check has a valid reference. This is
  consistent with P-15.2's first-turn behavior
  (`project-state-through-p15.2.md` §4: first-turn →
  `agentSessionLastAssistantOutput.text === ""` → novelty `no_reference`).
  **With a greeting seed, the first turn is no longer "first"** — it has the
  greeting as reference. This is a desirable behavior change (the greeting
  prevents the Writer from repeating the opening).

| ------ | ------------------------------ | --------------------------------- | --------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

- `cardId` path param: must match `^[0-9a-f]{64}$` (SHA-256 hex). Reject

- `greetingId`: must match `^g[0-9]+$`.

## K. Security Model

### K.1 No execution of imported content (absolute)

The following are **detected and blocked/preserved, never executed**:

- JavaScript / TypeScript / eval / Function constructor / EJS templates
- `getvar` / `setvar` / MVU variable execution
- `<script>` tags, HTML event handlers (`onload=`, `onclick=`)
- `import(...)` / `fetch(` / jQuery `$.load(` / remote plugin URLs
- Remote status bar / remote opening page
- Regex scripts (12 in test card) — metadata only
- Helper / extension scripts (7 in test card) — metadata only
- Auto-install commands

Each detected feature is recorded as a `CardImportBlockedFeatureV1` with
`status: "blocked"` (actively removed) or `"preserved-not-executed"` (left
in metadata but never run). The Import Report lists all of them.

### K.2 Content sanitization at parse time

- JSON is parsed with a depth limit (`MAX_CARD_JSON_DEPTH`) to prevent
  stack-overflow via pathological nesting.
- File size is checked **before** parse (`MAX_CARD_BYTES`).
- String fields are truncated to documented limits.
- No `eval`, no `vm.runInNewContext`, no `new Function`, no `child_process`
  anywhere in the import pipeline. The import is pure JSON.parse + field
  extraction + string cleaning.

### K.3 Path traversal prevention

- `cardId` is hex-only (SHA-256) — cannot contain `/`, `..`, `\`.
- `sourceFilename` is sanitized via `safeFilename()` and is metadata-only
  (never used in path construction).
- On-disk files have fixed names (`source.json`, `manifest.json`, etc.) —
  no user-controlled filenames reach the filesystem.
- `cardsDir` is resolved once at startup and is an absolute path; all card
  paths are `resolve(cardsDir, cardId, "<fixed-name>")`.

### K.4 No card body in Prompt / System Prompt / API

- Card body (character description, worldbook content beyond retrieved
  entries) never enters `System Prompt`, `Critic Prompt`, `Curator Prompt`,
  or `preset.text`.
- Only retrieved worldbook entries (via the existing retrieval pipeline)
  reach the Writer `context` port.
- Greeting content enters the session as a committed turn, not as a prompt
  template.
- API responses strip file paths (`[R]` fields) and separated raw fragments.

### K.5 Namespace isolation between cards

- Each card has a unique `cardId` → unique `worldbookResourceRef`
  (`card:<cardId>`).
- Each RP session has a unique `sessionId` (UUID) → unique worldbook scope
  (`session:<sessionId>:card:<cardId>`) → unique memory namespace
  (`rp-session:<sessionId>`).
- Two cards never share a worldbook scope or memory namespace. Two sessions
  of the same card have different `sessionId`s → different scopes (each
  gets its own seeded copy).
- The `AgentSessionStore` is keyed by `AgentSessionKeyV1` (which includes
  `conversationId = sessionId`) — different sessionIds never collide.

## P. Final Implementability Conclusion

### P.1 Constraints honored (checklist)

- [x] **No card script executed** — V1 detects and blocks all JS/eval/EJS/
      getvar/regex scripts/helper scripts/remote URLs. No `eval`, no `vm`,
      no `child_process`, no `new Function` in the import pipeline.
- [x] **No MVU implemented** — variable store, state updater, JSON Patch
      commit, EJS execution, getvar executor, status bar runtime, generic
      variable node, remote plugin sandbox are all NOT implemented. Variables
      are `detected` / `preserved` / `summarized` / marked
      `unsupported-runtime` only.
- [x] **No Writer/Critic/Critic Prompt modification** — `rp-runtime`,
      `agent-runtime/profileRegistry.ts`, and all prompt templates are
      untouched.
- [x] **No card hardcoded into project** — cards live in git-ignored
      `data/cards/`, never in `packages/`, `apps/`, `data/workflows/`, or
      docs. No real card body in Git. No card in Agent Profile / System
      Prompt / Workflow JSON.
- [x] **No real card body read** — this audit used only the structural
      summary from the task brief. No real card file was opened.
- [x] **Worldbook goes through existing data path** —
      `CardImport → worldbook.json → session-init seed →
DynamicWorldbookStore → dynamicWorldbook node → genericRetriever →
rpContextAssemblerV1 → Writer context`. No shortcut to Writer Prompt.
- [x] **No production code modified, no commit, no tag** — this document is
      a design report only.

### P.2 Verdict: **GREEN**

The existing infrastructure supports the full P-15.3A closed loop with
**no architectural blockers**:

1. **`DynamicWorldbookStore`** is reusable as-is — card import seeds it via
   `store.save()` at session init. No store interface change needed.
2. **`POST /api/rp`** accepts `worldbook.resourceRef` unchanged — a card
   resourceRef (`card:<cardId>`) flows through the existing pipeline with
   zero modification to the RP runtime, workflow JSON, or Writer/Critic.
3. **`AgentSessionStore`** supports greeting seeding via existing
   `store.append()` — no session type change needed (turn 0 with
   `assistantOutput = greeting`).
4. **Hono 4.12.25** has built-in multipart parsing (`c.req.parseBody()`) —
   no new dependency for file upload.
5. **Atomic file write + path-traversal-safe naming** patterns are
   established (`FileWorkflowCheckpointStore`) and reusable.
6. **Package boundary** (`@awp/card-import`) follows the `rp-runtime`
   precedent with a clean dependency direction.

**One yellow caveat**: the constant-entries behavior difference (O1) is a
deliberate V1 design choice mandated by the task, not a blocker. It is
documented and reversible via future workflow-level retrieval tuning.

**The design is ready for implementation in three batches (P-15.3A-1 →
P-15.3A-2 → P-15.3A-3) with no open architectural questions.**
