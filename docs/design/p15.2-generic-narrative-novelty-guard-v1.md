### Verified facts (from audit)

1. **`sessionLoad` output is structured JSON** with shape `AgentSessionContextV1 { turns: AgentTurnV1[] }`. Each `AgentTurnV1` carries `assistantOutput: string` of a **committed** assistant turn. Commit happens on `agentSessionCommitV1` for any turn whose `decision.allowSessionCommit === true` — that includes `exhausted-return-latest` (which still commits) and excludes `exhausted-fail` (which does not). See `packages/agent-runtime/src/rpSideEffectDecision.ts` for the policy.
2. **There is no `accepted` flag on `AgentTurnV1`**. The session store is a log of committed bodies, not a log of accepted bodies. The correct term is **previous committed** assistant output, never "previous accepted narrative".
3. **`writer2.data` is the existing revision-instruction channel**. `e_route_revise` carries the `condition` source's output into writer2. The `specializedAgent` executor renders the `data` port as a `## Data` markdown section. Today the source is `gate1.result`; P-15.2 will change the source to `merge1.decision` (see §6). No prompt template is edited.
4. **`finalDraftSelector` consumes three gates**: `acceptRouting`, `firstGateResult`, `secondGateResult`. Its `loopResult.exhausted` is set when `secondGateResult.accepted === false`. The selector does not read markdown and does not consult Writer-1 vs Writer-2 routing directly.
5. **`rpCriticQualityGate` input is `review` (critic output JSON)**. It does not receive the writer draft. It returns `{ accepted, revisionInstruction, failedChecks, issues, scores }`. **Critic `issues` are structured records** (per `awp.rp-critic-review.v1`), not `string[]`. P-15.2 must not flatten them to `string[]` — see §7.
6. **`rpSideEffectDecision` (existing) is the sole owner of `allowSessionCommit` / `allowPlayerOutput` / `allowMemoryCommit`**. It receives `loopResult` from `finalDraftSelector` and respects `behavior.onExhausted` (default `exhausted-return-latest`, alternate `exhausted-fail`). P-15.2 V1 does **not** modify this node. Novelty Merge only flags `accepted=false`; the existing `decision` node decides what "not accepted" means in terms of side effects.
7. **`specializedAgent` ports are `instruction`, `userInput`, `context`, `data`, `result`**. `data` is rendered as a `## Data` markdown block. Enriching this block is the natural revision-target injection point; no new instruction port, no preset change, no LLM-judge step.
8. **API/inspect contract** is driven by node IDs in `officialRpOutputAdapter` (extracts `narrative`, `quality`, `sessionCommit`, `memoryCommit` from specified node IDs). Inspect nodes are wired per-instance. Adding a `textNoveltyCheck` node + new inspect nodes does not change the public API response and does not leak the reference text.
9. **Curator is on the accepted-only path** (gated by `routeMemory` and `decision.allowMemoryCommit`). `exhausted-return-latest` allows session commit and player output but blocks memory commit; `exhausted-fail` blocks all three. P-15.2 inherits this as-is.
10. **Checkpoint/resume** is keyed by node ID. New nodes are checkpointed by reference. Restart-after-exhausted re-loads the same `sessionContext` and the same `turns[]` shape — the `agentSessionLastAssistantOutput` extraction remains correct.

### What must be added

- A node that extracts the previous committed assistant narrative from `sessionContext` (lives in `agent-runtime`, knows the session contract).
- A generic, RP-agnostic text-duplication checker (lives in `workflow-stdlib`).
- A `rpQualityDecisionMerge` node (lives in `agent-runtime`, knows the Critic Gate, revision, attempt, and exhausted semantics) that produces a **`decision` port for routing and a separate `diagnostics` port for inspect only**.

## 3. Algorithm Choice

### 3.1 Normalization (V1 — four deterministic steps, no language-specific mapping)

`normalize(text: string): string` performs, **in this exact order**:

1. `String.prototype.normalize('NFKC')` — collapses full-width/half-width, ligatures, and Unicode-compatibility forms into composed forms.
2. Strip Unicode BOM (`﻿`) and zero-width characters (`​`, `‌`, `‍`).
3. Replace every whitespace run (`\s+`) with a single ASCII space.
4. `String.prototype.trim()`.

**Explicitly NOT done in V1** (each was considered and rejected — see "rejected alternatives" below):

- Chinese punctuation folding (`，` → `,`, `。` → `.`, etc.)
- Quote folding (curly → straight or vice versa)
- Bracket / 书名号 folding
- Markdown marker stripping
- ASCII lowercasing
- Punctuation deletion
- Token-level Jaccard / n-gram similarity
- Embedding-based similarity
- LLM judge

**Rejected alternatives and the reason** (kept in the design for audit, not for re-opening):

| Alternative               | Reason rejected                                                                                                                                  |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| Strip Chinese punctuation | Would conflate `你来了?` with `你来了`; would conflate `他说："好。"` with `他说:好`; loses semantic distinction in RP dialogue                  |
| Strip quotes              | Loses dialogue boundaries that the player can perceive                                                                                           |
| Strip Markdown markers    | Would conflate "same text, different formatting" with "same text"; the player can perceive the marker difference                                 |
| ASCII lowercasing         | Chinese has no case; the only effect is on English fragments, and English lowercase is already a stylistic choice that the preset may care about |
| Punctuation deletion      | Same as Chinese punctuation — semantic loss                                                                                                      |
| n-gram / Dice / LCS       | Not needed for the observed 1/20 case; adds implementation surface; needs a labeled Chinese-RP corpus to calibrate thresholds; deferred to V1.1  |

The V1 algorithm is intentionally **language-neutral**: a French or Japanese preset uses the same normalize function with no change. Locale-specific rules belong in a V1.1 preset-config override (not in V1).

### 3.2 Comparison (V1)

```
normalizedCurrent := normalize(currentDraft)
normalizedReference := normalize(referenceText)

if referenceText is empty (turns.length === 0):
  report.reason = "no_reference"
  report.exactDuplicate = false
  report.evaluated = false
elif currentDraft is empty (after trim):
  report.reason = "empty_current"
  report.exactDuplicate = false
  report.evaluated = true
elif min(normal(normalizedCurrent.length, normalizedReference.length)) < minNormalizedLength:
  report.reason = "below_minimum_length"
  report.exactDuplicate = false
  report.evaluated = true
elif normalizedCurrent === normalizedReference:
  report.reason = "exact_duplicate"
  report.exactDuplicate = true
  report.evaluated = true
else:
  report.reason = "novel"
  report.exactDuplicate = false
  report.evaluated = true
```

Notes:

- `evaluated: true` means the comparison actually ran. `evaluated: false` means there was no reference to compare against.
- `below_minimum_length` skips the comparison but does not flag a duplicate; it defers to the Critic Gate.
- V1 does not iterate over a window of references; it uses the most recent committed turn only.
- **The revision string says "归一化后重复"**, not "byte-identical". The contract is normalized equality, and any future expansion of the normalization (even within V1) will widen the set of inputs the check catches — the contract remains "归一化后重复" throughout.

### 3.3 Threshold (`minNormalizedLength`)

Computed from the P-15.1 20-turn corpus (NFKC + collapse-whitespace + trim):

```
n=20   min=144   p25=168   median=202   p75=214   max=265   mean=192
```

| `minNormalizedLength` | Turns below threshold (out of 20) | Effect on real production data                       |
| --------------------- | --------------------------------- | ---------------------------------------------------- |
| 16                    | 0/20                              | no-op                                                |
| 64                    | 0/20                              | no-op (default)                                      |
| 80                    | 0/20                              | no-op                                                |
| 96                    | 0/20                              | no-op                                                |
| 128                   | 0/20                              | no-op                                                |
| 160                   | 5/20                              | trips on 5/20 real turns (rejected — too aggressive) |

**V1 default: `minNormalizedLength = 64`.** Rationale:

- All 20 real production bodies are ≥ 144 normalized chars. A threshold of 64 cannot mask any real duplicate.
- The threshold exists to defend against a pathological Writer returning a few characters (e.g., `""` → caught by `empty_current`; `"好"` → 1 char, well below 64). Such pathological outputs should be routed to the Critic for quality judgement, not the deterministic Novelty check.
- The threshold is configurable per node instance in the workflow JSON, so a future preset that legitimately produces short outputs can lower it without code changes.

### 3.4 Near-duplicate (V1.1 — not built, not wired, not specified)

The `reason` enum reserves no near-duplicate value. `exactDuplicate` is the only verdict field. No similarity score, no n-gram table, no threshold. **A V1 implementation that emits `nearDuplicate: false` with a fake similarity score of `0` is rejected** (cf. 必改 4).

## 4. Schema

All contracts are JSON-serializable, carry a stable `schemaId` string under the repo's `awp.*` namespace (verified convention: `awp.rp-critic-review.v1`, `awp.rp-revision-request.v1`, `awp.rp-revision-loop-result.v1`, `awp.dynamic-worldbook-result.v1`, etc.), and **never embed full reference text in the report**.

### 4.1 `awp.text-novelty-report.v1`

```typescript
export interface TextNoveltyReportV1 {
  schemaId: "awp.text-novelty-report.v1";
  evaluated: boolean; // false iff no_reference
  exactDuplicate: boolean; // only meaningful when evaluated === true
  normalizedCurrentLength: number;
  normalizedReferenceLength: number; // 0 when no_reference
  reason: "no_reference" | "empty_current" | "below_minimum_length" | "exact_duplicate" | "novel";
}
```

Notes:

- The runtime clock is **not** a field on the report. The `NodeRunResult` infrastructure (`startedAt`, `endedAt`) is the only place execution time lives. See §5 for the wiring.
- The `reason` enum is closed. Adding a value is a schema bump (`v2`).

### 4.2 `awp.rp-decision.v1` (the `decision` port of `rpQualityDecisionMerge`)

This is the **routing-grade** output: it carries only what `route`, `writer2.data`, and `finalDraftSelector` need.

```typescript
export interface RpDecisionV1 {
  schemaId: "awp.rp-decision.v1";
  accepted: boolean;
  decision: "accept" | "revise" | "exhausted";
  revisionInstruction?: string; // present iff decision === "revise"
  failedChecks: string[]; // short stable codes only
}
```

Notes:

- `failedChecks` is a `string[]` of stable codes (e.g., `exact_duplicate`, `critic_continuity_low`). It is **not** the structured Critic `issues` array — see §7 for how Critic issues are handled.
- `revisionInstruction` is length-capped (see §8) and lives in this object so Writer 2 receives it through the `data` port without seeing Gate scores, sources, or diagnostics.

### 4.3 `awp.rp-decision-diagnostics.v1` (the `diagnostics` port)

This is the **inspect-grade** output: it carries the full evidence chain. It is wired only to `inspectOutput` nodes; it never reaches `route`, `writer2`, or `selector`.

```typescript
export interface RpDecisionDiagnosticsV1 {
  schemaId: "awp.rp-decision-diagnostics.v1";
  attempt: 1 | 2;
  criticGate: {
    nodeId: string;
    accepted: boolean;
    failedChecks: string[];
    scores: Record<string, number>;
    issues: unknown[]; // verbatim structured Critic issues (preserved shape)
  };
  novelty: {
    nodeId: string;
    report: TextNoveltyReportV1;
  };
}
```

Notes:

- `issues` is `unknown[]` because the existing Critic `issues` payload is structured records per `awp.rp-critic-review.v1`; the diagnostics port passes them through unchanged rather than flattening to `string[]`. This resolves the 必改 7 type incompatibility by keeping each output type's native shape.

### 4.4 `awp.agent-session-last-assistant-output.v1` (the `text` port companion)

```typescript
export interface AgentSessionLastAssistantOutputV1 {
  schemaId: "awp.agent-session-last-assistant-output.v1";
  text: string; // empty when sessionContext.turns is empty
  turnCount: number;
  sourceTurnId?: string; // undefined when empty
  nodeId: string;
}
```

### 5.5 Execution time

`evaluatedAtMs` is **not** a field on the Novelty report. The existing `NodeRunResult.startedAt` / `NodeRunResult.endedAt` (already produced by the runner for every node execution) is the single source of execution-time telemetry. This guarantees bit-identical reports across runs (D1/D2 tests in §10.3 are now possible without a clock-stripping step).

### 6.3 Why this preserves all 13 design rules

1. ✅ ComfyUI-style explicit workflow and node data lines — three new node types, fully typed ports, edges in JSON.
2. ✅ Generic nodes — `textNoveltyCheck` is the only truly generic one (`workflow-stdlib`); the two RP-aware nodes are explicitly named `agentSessionLastAssistantOutput` and `rpQualityDecisionMerge` and live in `agent-runtime`.
3. ✅ Deterministic, no LLM.
4. ✅ No direct store access — only the existing `sessionLoad` output is consumed.
5. ✅ No full-session re-injection into Critic — Critic receives only `writer1.result` / `writer2.result` exactly as before.
6. ✅ No Critic prompt change.
7. ✅ Writer prompt unchanged in `instruction`; only Writer-2's `data` is enriched with the merged `decision.revisionInstruction`. Writer-1's `data` is unwired (unchanged).
8. ✅ Player agency, knowledge boundary, continuity hard gates untouched — `rpCriticQualityGate` thresholds and verdict logic unchanged.

## 7. Routing Semantics

### 7.1 `rpQualityDecisionMerge` V1 logic (precise)

```
function mergeAttempt({ gateResult, noveltyReport, attempt, noveltyRevisionInstruction }):
  accepted := gateResult.accepted
  failedChecks := [...gateResult.failedChecks]
  sources := []  // diagnostics only; not used in decision

  if noveltyReport.exactDuplicate:
    accepted := false
    failedChecks := [...failedChecks, "exact_duplicate"]
    if attempt === 1:
      revisionInstruction := noveltyRevisionInstruction
    else:
      // attempt === 2: same string; rpSideEffectDecision + behavior.onExhausted
      // decide what "not accepted" means. This node does not declare side effects.
      revisionInstruction := noveltyRevisionInstruction

  decision :=
    if accepted then "accept"
    else if attempt === 1 then "revise"
    else "exhausted"

  decisionPort := {
    schemaId: "awp.rp-decision.v1",
    accepted,
    decision,
    revisionInstruction: decision === "revise" ? revisionInstruction : undefined,
    failedChecks,
  }

  diagnosticsPort := {
    schemaId: "awp.rp-decision-diagnostics.v1",
    attempt,
    criticGate: {
      nodeId: gateResult.nodeId,
      accepted: gateResult.accepted,
      failedChecks: gateResult.failedChecks,
      scores: gateResult.scores,                  // kept here; never reaches Writer
      issues: gateResult.issues,                  // structured records; never flattened
    },
    novelty: {
      nodeId: noveltyReport.nodeId,
      report: noveltyReport,
    },
  }

  return { decision: decisionPort, diagnostics: diagnosticsPort }
```

### 7.2 Why the `decision` and `diagnostics` split

- `decision` is the only port that flows to `route.condition`, `writer2.data`, and `selector.secondGateResult`. It is small (≤ 4 fields, ≤ ~200 bytes for the revision string).
- `diagnostics` is the only port that flows to `inspMerge*`. It carries the full evidence chain (Critic scores, structured issues, novelty report, attempt).
- **No edge connects `diagnostics` to `writer2`, `route`, or `selector`.** This is enforced by the JSON wiring; an audit script (§10.5 R5) verifies it.
- `revisionInstruction` is length-capped at 200 UTF-8 bytes in code (defense in depth) so a future workflow-author cannot accidentally inflate Writer 2's prompt and regress P-15.1's prompt-trimming property.

### 7.3 Routing table (V1)

| Attempt | Novelty                                                            | Critic | `merge.decision` | Route / Selector path                                                                            | Writer 2 invocation | Side effects (per `rpSideEffectDecision`)                                                       |
| ------- | ------------------------------------------------------------------ | ------ | ---------------- | ------------------------------------------------------------------------------------------------ | ------------------- | ----------------------------------------------------------------------------------------------- |
| 1       | pass (novel / no_reference / below_minimum_length / empty_current) | accept | `accept`         | `route.acceptBranch` → `selector.acceptDraft = writer1`                                          | no                  | accepted path: commit + memory-policy (per existing rules)                                      |
| 1       | pass                                                               | revise | `revise`         | `route.reviseBranch` → `writer2` with Critic's `revisionInstruction`                             | yes                 | if Writer 2 accepted, commit + memory-policy; if exhausted, `behavior.onExhausted`              |
| 1       | **exact_duplicate**                                                | accept | `revise`         | `route.reviseBranch` → `writer2` with **novelty's** `revisionInstruction`                        | yes                 | (same)                                                                                          |
| 1       | **exact_duplicate**                                                | revise | `revise`         | `route.reviseBranch` → `writer2` with novelty's `revisionInstruction` (concatenation rule below) | yes                 | (same)                                                                                          |
| 2       | pass                                                               | accept | `accept`         | `selector` accepts Writer 2's draft                                                              | no (already ran)    | accepted path                                                                                   |
| 2       | pass                                                               | revise | `exhausted`      | `selector.secondGateResult.accepted = false` → `loopResult.exhausted = true`                     | no                  | **`behavior.onExhausted` decides side effects** (existing policy; P-15.2 V1 does not change it) |
| 2       | **exact_duplicate**                                                | accept | `exhausted`      | same as above                                                                                    | no                  | same as above                                                                                   |
| 2       | **exact_duplicate**                                                | revise | `exhausted`      | same as above                                                                                    | no                  | same as above                                                                                   |

**Concatenation rule (when both novelty and critic are hard-fail):** V1 emits the **novelty** `revisionInstruction` (from workflow JSON config) **only**. The Critic's `revisionInstruction` is preserved in `diagnostics.criticGate` for inspect. The reason: when both are present, novelty is the dominant cause (it means "the draft is the same as the last turn"); the Critic's revision is the older, generic instruction and would dilute the prompt if concatenated. This is a V1 simplification, not a V2 deprecation; a future preset may opt in to concatenation via a `revisionInstructionMode` config flag (deferred).

### 7.4 First-turn behavior

- `sessionContext.turns.length === 0`
- `agentSessionLastAssistantOutput.text === ""`
- `textNoveltyCheck.report.reason = "no_reference"`, `evaluated = false`
- `merge.decision.accepted` follows the Critic gate → first turn is unaffected.

### 7.5 No third attempt is introduced

- Attempt 1 hard-fail (novelty or critic) → revise → Writer 2.
- Attempt 2 hard-fail (novelty or critic) → `exhausted` → `rpSideEffectDecision` decides side effects.
- `merge2.decision === "exhausted"` is informational; the actual `loopResult.exhausted` is determined by `finalDraftSelector` reading `secondGateResult.accepted === false`.

## 8. Revision Prompt Integration

### 8.1 Existing channel (verified)

`writer2.data` receives `route.reviseBranch`, which is the JSON output of whatever sits on the `condition` input of `route`. Today that is `gate1.result` (Critic gate output). After the V1 wiring change, it becomes `rpQualityDecisionMerge1.decision` (`awp.rp-decision.v1`):

```json
{
  "schemaId": "awp.rp-decision.v1",
  "accepted": false,
  "decision": "revise",
  "revisionInstruction": "本轮正文与上一轮正文重复。请根据玩家最新输入重写并明确推进剧情，不要复用上一轮正文。",
  "failedChecks": ["exact_duplicate"]
}
```

The `specializedAgent` executor renders the `data` port as a `## Data` section. **No prompt template is edited.** The Writer 2 prompt sees: `instruction = preset` (style), `userInput = input.text` (player), `context = ctxMerge2.result` (worldbook + memory + session-md), `data = merge1.decision` JSON.

### 8.2 Instruction is in the workflow JSON, not in production code (Rev 2)

The V1 revision string lives in `rpQualityDecisionMerge{1,2}.config.noveltyRevisionInstruction`. The default in the official workflow JSON is:

```
本轮正文与上一轮正文重复。请根据玩家最新输入重写并明确推进剧情，不要复用上一轮正文。
```

Rationale:

- **Visible to the workflow author** — diffs to the JSON are the single place to change the instruction.
- **No bilingual duplication** — the official RP is Chinese, so a single Chinese string avoids the prompt-token bloat that Rev 1's bilingual form would have caused.
- **A future English preset** can override the field with an English string without code changes.
- **Length cap** — `rpQualityDecisionMerge` rejects a `noveltyRevisionInstruction` longer than 200 UTF-8 bytes (defense in depth; well above the V1 default of ~50 bytes, well below the Writer prompt budget).

### 8.3 Critic 2 does not see the Novelty report

`critic2.instruction` is built by `criticInstructionBuilder` from `criticRubric` + `gate1.result`. P-15.2 does not alter `criticInstructionBuilder` or its inputs. The Critic 2 prompt is unchanged. The Novelty report is observable via `inspNovelty2` (json) but does not influence Critic 2's verdict. This is a deliberate separation of concerns: Critic-2 scores a candidate draft against the rubric; Novelty is the deterministic correctness check.

### 8.4 Writer 2 prompt does not contain Gate scores, sources, or Novelty diagnostics

The `diagnostics` port of `rpQualityDecisionMerge{1,2}` is **not** wired to `writer2.data` (or to `route` or to `selector`). Writer 2 sees only the `decision` port's four fields. Verified by the wiring diagram in §6.2 and the R5 audit test in §10.5.

## 9. Side Effects and Checkpoint

### 9.1 Session commit and Memory commit

`rpSideEffectDecision` (existing, unchanged) is the **sole owner** of `allowPlayerOutput` / `allowSessionCommit` / `allowMemoryCommit`. Its policy is:

| `loopResult.exhausted` | `behavior.onExhausted`              | `allowPlayerOutput` | `allowSessionCommit`                            | `allowMemoryCommit`                        |
| ---------------------- | ----------------------------------- | ------------------- | ----------------------------------------------- | ------------------------------------------ |
| `false` (any reason)   | (n/a)                               | `true`              | `true`                                          | `true` (subject to `rpMemoryCommitPolicy`) |
| `true`                 | `exhausted-return-latest` (default) | `true`              | **`true`** (commits the latest available draft) | **`false`**                                |
| `true`                 | `exhausted-fail`                    | `false`             | `false`                                         | `false`                                    |

**P-15.2 V1 does not add a new side-effect policy.** When a Writer-2 novelty-exact-duplicate produces `exhausted`:

- If `behavior.onExhausted === "exhausted-return-latest"` (default): the player still receives the latest available draft; a Session commit still happens; memory is not written. **The duplicate draft is not promoted to a commit** because `secondGateResult.accepted === false` causes `finalDraftSelector` to fall back to the previous accepted draft or, if absent, to fail the workflow — see `finalDraftSelector` logic. The "latest available" draft in this scenario is therefore the prior committed one, not the duplicated Writer-2 draft.
- If `behavior.onExhausted === "exhausted-fail"`: the workflow fails; no commit, no memory.

**P-15.2 V1 does NOT claim "exhausted ⇒ no session commit" as a rule.** That statement was Rev 1; it was wrong because it conflated Novelty Merge with `rpSideEffectDecision`. The correct framing is: Novelty Merge sets `accepted = false` on the Writer-2 path; `finalDraftSelector` derives `loopResult.exhausted = true`; `rpSideEffectDecision` then enforces the configured `behavior.onExhausted`. If a future change wants "novelty-exhausted ⇒ no commit", that is a new `rpSideEffectDecision` policy — explicitly out of scope for P-15.2 V1.

### 9.2 Idempotency

Re-running a turn with the same inputs produces the same `agentSessionLastAssistantOutput.text` (since `sessionContext` is read-only between runs) and the same `textNoveltyCheck.report` (the report is now clock-free, see §5.5). `rpQualityDecisionMerge` is a pure function of its inputs. No regression risk vs P-15.1.

### 9.3 Memory commit

- `exhausted-return-latest` does not write long-term memory (existing behavior, preserved). The Player receives a draft; the session log gets a new committed turn; memory is untouched.
- `exhausted-fail` writes neither session nor memory.
- A non-exhausted novelty-revised turn (Writer 2 produced a fresh, non-duplicate draft, accepted by the Critic) does run Curator. Correct: there is real new content worth curating.

### 9.4 Checkpoint / resume

- New nodes are checkpointed by node ID. The merged `decision` JSON and the `diagnostics` JSON together are bounded (~1 KB per attempt, summed across 2 attempts is ~2 KB per turn).
- Restart-after-exhausted: `sessionLoad` re-emits the same `sessionContext`. `agentSessionLastAssistantOutput` returns the same text. `textNoveltyCheck` reproduces the same report. `rpQualityDecisionMerge` reproduces the same `decision` and `diagnostics`. No drift.
- Restart-mid-turn (e.g., after merge1 but before writer2): on resume, merge1 is re-executed (pure function of checkpointed inputs) and writer2 is invoked with the same `data` payload. No double-counting of attempts (the `attempt` field is set by config, not by Writer invocation count).

### 9.5 Telemetry

- `NodeRunResult.startedAt` / `NodeRunResult.endedAt` (existing) carry execution time for all three new node types.
- The `diagnostics` port is the only path that exposes Gate scores, structured Critic issues, and the full Novelty report to observability. The `decision` port never carries these.
- The API response (extracted by `officialRpOutputAdapter`) is unchanged in field set. `textNoveltyCheck` and `rpQualityDecisionMerge` do not introduce new public fields.

## 14. Out of Scope (explicit)

1. **Semantic "did the player instruction get followed?" checks** — out of scope. Critic's `playerAgency` score is the existing channel.
2. **Embedding / vector similarity / LLM judge** — explicitly excluded.
3. **Near-duplicate as a hard revise trigger** — V1.1, contingent on a labeled corpus and threshold calibration.
4. **Auto-curating novelty reports into long-term memory** — Curator sees only the final accepted draft, unchanged.
5. **Window > 1 for reference text** — V1 uses the most recent committed turn only.
6. **Multi-locale normalization (Japanese kana folding, Arabic diacritics, etc.)** — V1 is language-neutral by design; locale-specific rules are a V1.1 preset-config override.
7. **Per-preset normalization rules** — V1 has a single global `normalize` function. Per-preset overrides are V1.1.
8. **A third Writer / Critic pass** — excluded.
9. **Removing the existing `renderRevisionPrompt` from `rpRevisionLoop.ts`** — preserved for backward compatibility; not used in the V1 hot path.
10. **Changing Writer-1's prompt** — Writer-1's `data` port is unwired; P-15.2 leaves it unwired.
11. **A new `rpSideEffectDecision` policy for "novelty-exhausted"** — P-15.2 V1 does not add it. The existing `behavior.onExhausted` is the only exhausted-side-effect policy. If the user wants a stricter "novelty-exhausted ⇒ no commit" rule, that is a separate ticket.
12. **Bilingual revision instructions** — V1 official RP is Chinese; a single Chinese string is the default. A future English preset overrides the workflow-JSON field.
13. **Lowering `minNormalizedLength` below 64** — V1 default is 64. Workflow authors may lower it; that is a workflow-author choice, not a code change.

## 15. Open Questions (must be resolved before implementation)

These are deliberate deferrals. Each is small and answerable; none blocks the design from being approved.

1. **Q1 — Inspect node placement for merge1/merge2**: should they be auto-added (always visible) or opt-in? Recommendation: **opt-in via workflow JSON**, matching the existing convention. The base workflow JSON should add them to match P-15.1's pattern of inspecting every gate.
2. **Q2 — `minNormalizedLength=64` as the V1 default**: see §3.3. Recommendation: **64** — well below the real-data minimum of 144, acts as a safety net only.
3. **Q3 — Should the `awp.text-novelty-report.v1` schema include a `schemaVersion: 1` field, or is `schemaId` enough?** Recommendation: `schemaId` only; matches the existing repo convention (`awp.rp-critic-review.v1` does not carry a `schemaVersion`).
4. **Q4 — Default `noveltyRevisionInstruction` text**: see §8.2. Recommendation: the V1 default listed in §6.2.
5. **Q5 — Should `agentSessionLastAssistantOutput` also expose the second-most-recent committed turn as a fallback reference for restart cases where the most recent turn was just committed by the same call?** Recommendation: **no** in V1; V1 uses only the most recent. Windowing is a V1.1 concern.
6. **Q6 — Anti-false-positive fixture AF1** (fixed-line duplication): ship as a known-limitation test (V1 expects to flag it) or backport a "do not compare quotations only" rule? Recommendation: **ship as known-limitation test in V1**; the V1.1 fixture set is the right place to fix it once we have a labeled corpus.
7. **Q7 — `rpQualityDecisionMerge.attempt` config field, or infer from `gateResult` shape?** Recommendation: **explicit `attempt`** (1 or 2). Inferring is brittle and couples the merge node to the gate node's output schema.
8. **Q8 — Naming `agentSessionLastAssistantOutput`**: kept in Rev 2 because it is the most explicit about being a session extractor. Alternatives (`sessionLastAssistantOutput`, `lastCommittedAssistantOutput`) are acceptable if preferred; the design is agnostic to the exact spelling.
9. **Q9 — Is the `noveltyRevisionInstruction` text allowed to be edited via a per-preset config, or only via the workflow JSON?** Recommendation: **workflow JSON only in V1**. Preset-configurable instruction strings are a V1.1 concern.
10. **Q10 — Should the `diagnostics` port be optional on `rpQualityDecisionMerge`, or always present?** Recommendation: **always present**; the audit script (R5) treats it as a required output. Workflow authors may leave it unwired in the JSON.
11. **Q11 — Should the P-15.1 20-turn corpus be re-run as part of the V1 validation, or only the deterministic-replay fixtures?** Recommendation: **both**. The corpus re-run is the regression check; the deterministic-replay fixtures are the correctness check.
