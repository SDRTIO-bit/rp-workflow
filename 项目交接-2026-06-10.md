## 设计/产品约定

当前约定如下：

- 不引入 React Flow，继续自研画布。
- 不改变现有 workflow JSON 结构。
- 数据类型连接严格阻止，不自动插入转换节点。
- Agent 可以有多个，由用户决定数量。
- Agent 独立存在，可以并行处理。
- 缓存命中价格很低，后续设计 Agent 调用时要尽量利用缓存命中，避免无意义重复上下文变动。
- 继续以中文为主要界面语言，保留英文切换。

## 外来扩展节点开发规范

本轮已把节点体系整理为“registry 驱动”的扩展方式。新增节点优先改
`packages/workflow-core/src/nodeRegistry.ts`，不要在前端单独硬编码节点清单。

每个节点定义建议包含：

- `type`：稳定机器名，作为 workflow JSON 的节点类型，不要随意改名。
- `label`：英文默认显示名。
- `category`：节点分组，当前使用 `core`、`knowledge`、`external`、`utility`。
- `description`：节点用途说明，会显示在检查器预览区。
- `color`：节点主题色，前端会直接读取。
- `preview`：节点输入输出摘要，用于节点卡片和检查器预览。
- `defaultConfig`：新增节点时的默认配置。
- `ports`：输入输出端口，端口数据类型必须来自 `DataType`。

如果新增了新的 `DataType`，必须同步更新：

- `packages/workflow-core/src/types.ts`
- `packages/workflow-core/src/nodeRegistry.ts` 里的 `areTypesCompatible`
- `apps/web/src/portPresentation.ts`

如果该节点需要本地 mock 运行结果，必须同步更新 `apps/web/src/App.tsx` 里的
`runLocalWorkflow` executor；真实服务端执行逻辑后续应放到服务端脚本或 runtime 包中。

## 后续建议

建议下一个对话优先做这些：

1. 给画布交互补浏览器自动化测试，比如真正模拟拖线、缩放、平移。
2. 增加端口多连接策略配置，目前目标输入端口统一按单输入阻止重复连接。
3. 给 workflow 保存到服务端或文件库做版本管理。
4. 给 Agent 调用增加更清晰的缓存命中显示，例如每个节点显示 cache hash、命中/未命中、token 估算。
5. favicon 可顺手补一个，消除浏览器 404。
6. 给外来扩展节点补真实 executor 和端到端浏览器测试。
